## 배경: 왜 하네스를 만들게 됐나

ProcessLens라는 제조 공정 분석 SaaS를 Claude Code로 만들고 있다. FastAPI + Streamlit + Supabase 기반으로, 배터리/화학 제조 엔지니어들이 공정 데이터에서 수율 손실 원인을 찾을 수 있게 돕는 플랫폼이다.

ProcessLens를 만들면서 자연스럽게 9개의 문서 체계(exec-plans, product-specs, ARCHITECTURE.md, AGENTS.md 등)를 갖추게 됐다. FOMO 1주차 발표를 듣고 나서야 깨달았다 — **이게 하네스 일수도 있겠다.**

그런데 문제가 있었다. 이 문서 체계는 ProcessLens 전용이다. 새로운 프로젝트를 시작하면? 처음부터 다시 만들어야 한다. 위스키 테이스팅 노트 앱을 만들고 싶으면, 주식 자동매매 시스템을 만들고 싶으면 — 매번 같은 삽질을 반복해야 한다.

**"어떤 프로젝트에서든 작동하는 범용 하네스를 만들 수는 없을까?"**

이 질문에서 JStack이 시작됐다. 아마 밑에 틀린 내용이 많을 수도 있지만 차근차근 정리해보고 공부해보고자 한다. 
비개발자의 도전기라고 보면 좋을 것 같다.

## G Stack에서 받은 영감

현재 사이드 프로젝트로 G Stack(Garry Tan의 Claude Code 하네스)을 직접 사용해보고 공부하고 있다. G Stack의 `/office-hours` SKILL.md를 517줄 전문으로 읽으면서 많은 영감을 받았다.

### Forcing Questions 패턴

그냥 "사용자가 누구예요?"라고 묻는 게 아니다. 구조가 있다:

- **Ask**: 도전적이고 구체적인 질문을 던지고
- **Push until**: 만족스러운 답이 나올 때까지 밀어붙이고
- **Red flags**: 나쁜 답변 패턴이 보이면 경고한다

예를 들어 Q3(Desperate Specificity)은 이렇게 물어본다: *"이것이 가장 필요한 실제 사람의 이름을 대보세요. 직함이 뭐예요?"* — 카테고리 수준 답변("헬스케어 기업")이 나오면 Red flag로 잡는다.

### Operating Principles + Anti-slop

단순히 "CEO처럼 생각해"가 아니라, 어떤 원칙으로 판단하는지를 명시한다. 그리고 AI가 빠지기 쉬운 "칭찬 슬롭"을 Good/Bad 예시로 차단한다.

### 프롬프트 밀도의 중요성

G Stack은 25개 스킬에 517줄짜리 깊은 프롬프트를 쓴다. OMC는 32개 에이전트 + 40개 스킬이지만 각 프롬프트는 ~50줄로 얕다. **적은 수의 깊은 프롬프트가 많은 수의 얕은 프롬프트보다 효과적이라는 걸 배웠다.**

## JStack: 내가 만든 하네스

G Stack의 프롬프트 깊이 + OMC의 오케스트레이션 아이디어를 결합해서, 외부 의존성 없이 마크다운 파일만으로 작동하는 4계층 범용 하네스를 설계했다.

### 구조

```
L4 통제:  CLAUDE.md + rules/           ← 항상 작동
L3 실행:  commands/ (5단 기어)          ← 호출 시만 로딩
L2 품질:  settings.json + hooks        ← 항상 작동
L1 구조:  CHECKPOINT.md + AUDIT.log    ← 항상 작동
```

### 5단 기어 워크플로

```
/office-hours → /plan → /build → /review → /ship
```

| 기어 | 역할 | HARD GATE |
|------|------|-----------|
| /office-hours | 아이디어 검증 + 디자인 문서 생성 | 코드 작성 절대 금지 |
| /plan | 아키텍처 설계 + 숨은 가정 감사 | 애플리케이션 코드 금지 |
| /build | 태스크 큐 기반 순차 구현 | 아키텍처 변경 금지 |
| /review | paranoid staff engineer 코드 리뷰 | 테스트 실패 시 승인 불가 |
| /ship | 배포 + 릴리즈 노트 + 필수 회고 | /review 미통과 시 배포 불가 |

### G Stack / OMC와 비교

| 지표 | G Stack | OMC | JStack |
|------|---------|-----|--------|
| 크기 | 150KB+ | 300KB+ | 44KB |
| 외부 의존성 | Bun, Playwright | npm, tmux | 없음 |
| 프로젝트 자동 생성 | ✕ | ✕ | ✅ |
| 아키텍처 보호 | ✕ | △ | ✅ |
| 스코프 크리프 방지 | ✕ | ✕ | ✅ |
| CHECKPOINT 상태머신 | ✕ | ✕ | ✅ |

### JStack만의 차별점

1. **프로젝트 CLAUDE.md 자동 생성** — /office-hours가 질문을 던지고, 답변을 바탕으로 프로젝트 전용 하네스를 자동 생성
2. **CHECKPOINT.md 상태머신** — 세션이 끊겨도 다음 세션에서 자동으로 맥락 복원
3. **복잡도 게이트** — Simple/Medium/Complex 자동 분기
4. **아키텍처 보호 + 스코프 크리프 방지** — /build에서 물리적으로 차단
5. **제로 의존성** — 마크다운 파일만으로 작동

## 이번 주 배운 것

### 1. 하네스 = AI 에이전트의 OS
단일 프롬프트가 아니라, Rules + Context + Skills + Sandbox + Hooks + Evaluation + Sub-agents + Memory — 8대 구성요소의 결합이다.

### 2. 제약이 품질을 만든다
"하지 마"를 명시하는 HARD GATE, allowed-tools 제한, NEVER 규칙이 프롬프트의 핵심이다. AI는 "해라"보다 "하지 마라"에 더 잘 반응한다.

### 3. 상태 전달은 파일 시스템이 답이다
세션 메모리에 의존하면 세션이 끊길 때 맥락이 사라진다. CHECKPOINT.md에 상태를 기록하면 어떤 세션에서든 이어서 작업할 수 있다.

### 4. 범용 하네스 = "좋은 질문의 체계"
모든 도메인을 아는 하네스가 아니라, 어떤 도메인이든 올바른 질문을 던지고 구조화하는 하네스. 도메인 지식은 사람의 머릿속에서 나오고, 하네스는 그걸 끌어내는 역할이다.

### 5. Rippable하게 만들어라
모델이 진화하면 불필요한 레이어를 떼어낼 수 있도록 모듈식으로 설계해야 한다.

## 다음 주 계획

### 주식 자동매매 시스템으로 실전 테스트

다음 주부터는 **주식 자동매매 시스템**을 JStack 하네스 위에서 만들면서, 하네스의 실전 효과를 검증할 계획이다.

- `/office-hours`로 자동매매 시스템의 아이디어를 검증하고
- `/plan`으로 백테스트 아키텍처를 설계하고
- `/build`로 구현하면서 하네스가 실제로 어떻게 작동하는지 체감

이 과정에서 **하네스의 변경이 필요한 부분을 발견하고 수정**해나가고자 한다. 하네스는 한 번 만들고 끝이 아니라, 실전에서 쓰면서 다듬는 것이니까.
