## 배경: 왜 하네스 대신 실무를 택했나

1주차에 JStack이라는 범용 하네스를 설계했다. 그런데 하네스를 계속 다듬는 것보다, 실제로 내가 필요한 도구를 만들면서 하네스적 사고를 적용하는 게 더 의미 있겠다고 판단했다.

나는 배터리 셀 제조 수율 분석 SaaS인 **ProcessLens**를 만들고 있다. 이 프로젝트에서 가장 큰 문제 중 하나는 **분석 엔진을 검증할 데이터가 없다는 것**이었다. 실제 공장 데이터는 NDA 때문에 못 쓰고, 그렇다고 랜덤 데이터를 만들면 공정 물리를 반영하지 못한다.

그래서 **Synth-Analyzer**라는 사이드 프로젝트를 만들었다. 배터리 제조 공정의 물리 법칙을 반영한 합성 데이터를 생성하고, 여러 분석 알고리즘을 돌려서 최적의 분석 방법을 자동으로 찾아주는 도구다.

이걸 만들면서 "완전 자동화"를 목표로 잡았는데, 자동화 파이프라인을 설계하다 보니 자연스럽게 하네스의 구조가 필요해졌다.

## 만든 것: Synth-Analyzer

### 전체 파이프라인

```
[스키마 정의] → [데이터 생성] → [벤치마크 분석] → [GAN 최적화] → [최적 레시피 추천]
```

하나씩 설명하면 이렇다.

### 1단계: 스키마 — 도메인 지식의 구조화

배터리 셀 제조 공정을 12개 공정 단계, 126개 변수로 정의한 엑셀 스키마(`battery_cell_manufacturing.xlsx`)를 만들었다.

```
슬러리 믹싱 (15개) → 전극 코팅 (13개) → 건조 (12개) → 캘린더링 (12개)
→ 슬리팅 (9개) → 셀 조립 (12개) → 전해액 주입 (10개) → 포메이션 (12개)
→ 에이징 (8개) → 최종 검사 (10개) → 환경 (5개) → 원자재 (8개)
```

각 변수마다 **단위, 범위(Min/Max), 중심값, 표준편차, 분포 타입(normal/lognormal/uniform), 카테고리(equipment/process/material/quality)**를 정의했다.

이건 논문에서 하나하나 발췌한 값이다. 예를 들어:

| 변수 | 단위 | Min | Max | 분포 | 근거 |
|------|------|-----|-----|------|------|
| 믹싱 온도 | °C | 20 | 40 | normal | 슬러리 겔화 방지 범위 |
| 고형분 함량 | wt% | 40 | 80 | normal | NMC 양극 기준 |
| 슬러리 점도 | cP | 1,000 | 20,000 | lognormal | Krieger-Dougherty 모델 |

### 2단계: 상관관계와 물리 공식

변수들이 독립적이면 현실적인 데이터가 아니다. 그래서 **30개 상관관계**와 **5개 물리 공식**을 정의했다.

상관관계 예시:

```
믹싱 온도 ──(-0.7)──→ 슬러리 점도       (Arrhenius: 온도↑ → 점도↓)
고형분 함량 ──(+0.85)──→ 슬러리 점도     (Krieger-Dougherty: 고형분↑ → 점도↑)
캘린더링 압력 ──(-0.85)──→ 공극률         (압축 역학)
코팅 속도 ──(+0.6)──→ 코팅 결함률        (속도↑ → 결함↑)
```

물리 공식 예시:

```python
# 슬러리 점도 = Arrhenius + Krieger-Dougherty
η = A × exp(Ea/(R×(T+273.15))) × (1 - φ/φ_max)^(-[η]×φ_max)

# 코팅 로딩 = 밀도 × 두께
loading = ρ_slurry × h_wet × solid_content / 100

# 셀 용량
Q = loading × AM_ratio/100 × specific_cap / 1000 × area
```

이 상관관계와 공식들은 논문에서 발췌했고, **내가 직접 검수했다**. 배터리 공정 엔지니어로서 "이 상관관계가 현실에서도 맞는가"를 하나하나 확인하는 과정을 거쳤다.

### 3단계: 데이터 생성 — Cholesky 3계층 파이프라인

스키마와 상관관계를 바탕으로 합성 데이터를 생성한다. Cholesky 분해 기반 3계층 파이프라인으로 구현했다.

```
1계층: Cholesky 분해 → 상관 구조를 가진 다변량 정규분포 생성
2계층: Marginal Transform → 각 변수를 정의된 분포(lognormal, uniform 등)로 변환
3계층: 물리 공식 적용 → 공식으로 연결된 변수들의 관계를 물리적으로 보정
```

결과적으로 **205개 변수 × 만 행 이상**의 합성 데이터가 나온다. 여기에 NG(불량) 데이터도 비율에 맞춰 생성되며, 어떤 변수가 NG의 원인인지(ground truth)도 함께 기록된다.

### 4단계: 벤치마크 — 어떤 분석이 가장 잘 찾는가

생성된 데이터를 가지고 4가지 CPU 분석 + 4가지 GPU 분석을 돌린다.

**CPU 분석:**
- 상관분석 (Point-Biserial) — 이진 결과와 변수 간 상관계수
- 이상치 탐지 (Isolation Forest) — NG 데이터와 이상치 겹침
- 변수 중요도 (SHAP) — 변수별 영향도 순위
- 분포 비교 (KS/KL) — OK와 NG 간 분포 차이

**GPU 분석:**
- AutoEncoder, TabNet, XGBoost, DeepSHAP

ground truth를 알고 있으니까, 각 분석이 실제 NG 원인을 얼마나 잘 찾는지 **정량적으로 비교**할 수 있다.

### 5단계: GAN 최적화 — 난이도를 올려가며 스트레스 테스트

```
easy(신호 명확) → medium → hard → extreme(신호 거의 없음)
```

| 난이도 | 이상치 크기(σ) | 원인 변수 수 | 노이즈 | 원인불명 비율 |
|--------|:---:|:---:|:---:|:---:|
| easy | 3.0 | 1 | 0% | 20% |
| medium | 2.0 | 2 | 10% | 40% |
| hard | 1.5 | 3 | 20% | 60% |
| extreme | 1.0 | 4 | 30% | 80% |

각 난이도에서 데이터 생성 → 분석 → 레시피 추천을 반복하면서, **어떤 난이도에서 어떤 분석이 가장 효과적인지** 자동으로 탐색한다.

## 하네스적 사고의 적용

이 파이프라인을 만들면서 깨달은 건, **데이터를 만드는 과정에도 하네스의 구성 요소가 자연스럽게 들어간다**는 것이다.

### 적용된 하네스 요소

#### Rules — 스키마가 곧 규칙이다

JStack에서 "아키텍처 변경 금지", "/build에서 스코프 크리프 방지" 같은 NEVER 규칙을 넣었듯이, Synth-Analyzer에서는 **스키마가 규칙**이다.

```
믹싱 온도는 20~40°C만 허용 (normal 분포)
고형분 함량은 40~80wt%만 허용 (normal 분포)
슬러리 점도는 lognormal 분포를 따라야 함
```

생성기는 이 규칙을 벗어나는 데이터를 만들 수 없다. 코드 하네스의 "이 파일을 수정하지 마라"와 같은 역할이다.

#### Context — 논문이 곧 컨텍스트다

JStack에서 `CHECKPOINT.md`에 프로젝트 맥락을 기록하듯이, Synth-Analyzer에서는 **논문에서 발췌한 물리 모델이 컨텍스트**다.

`correlations.csv`의 30개 상관관계와 `formulas.csv`의 5개 물리 공식이 "이 데이터가 왜 이런 형태여야 하는지"를 설명한다. Arrhenius 방정식, Krieger-Dougherty 모델, Washburn 방정식 — 이런 물리 법칙이 생성기의 "도메인 지식"으로 주입된 것이다.

#### Skills — 탭이 곧 스킬이다

JStack의 5단 기어(`/office-hours → /plan → /build → /review → /ship`)처럼, Synth-Analyzer도 5개 탭이 순서대로 워크플로를 구성한다.

```
Tab 1 데이터 생성  ≈ /build    (구현)
Tab 2 벤치마크     ≈ /review   (검증)
Tab 3 GPU 분석     ≈ /review   (심층 검증)
Tab 4 GAN 최적화   ≈ /ship     (최종 산출물)
Tab 5 상세 결과    ≈ 회고       (결과 분석)
```

#### Evaluation — GAN 최적화가 곧 평가다

JStack에서는 "테스트 통과 여부"가 평가 기준이었다. Synth-Analyzer에서는 **4단계 난이도별 탐지 점수**가 평가 기준이다. easy에서 extreme까지 난이도를 올려가면서 "이 분석 방법이 현실의 어려운 상황에서도 작동하는가"를 자동으로 검증한다.

#### Human-in-the-loop — 도메인 전문가 검수

스키마의 변수 값, 상관관계 계수, 물리 공식의 적용 방식을 내가 직접 검토했다. `battery_cell_manufacturing_review.md`에 검수 결과를 기록하고, 피드백을 반영해서 스키마를 수정했다. AI가 논문에서 뽑은 값을 사람이 검증하는 구조다.

### 아직 없는 것 — 현재 구현 중

#### HARD GATE: 데이터 품질 게이트 (구현 중)

현재는 데이터를 생성한 뒤 바로 벤치마크로 넘어갈 수 있다. 생성된 데이터가 물리적으로 말이 안 되는 상태여도 분석을 시작할 수 있다는 뜻이다.

도입하려는 게이트:

```
데이터 생성 완료
    → [자동 검증]
       ✅ 상관관계 방향이 correlations.csv와 일치하는가?
       ✅ 물리 공식 변수들(formulas.csv)의 관계가 성립하는가?
       ✅ NG 비율이 설정값 ±허용오차 내인가?
    → 통과 시: 벤치마크 버튼 활성화
    → 실패 시: 벤치마크 버튼 비활성화 + 어떤 검증이 실패했는지 표시
```

JStack의 "/review 미통과 시 /ship 배포 불가"와 같은 패턴이다. 에러가 나서 못 가는 것과, 게이트가 막아서 못 가는 것은 다르다. **"못 가게 하는 것"이 아니라 "가면 안 되는 이유를 알려주는 것"**이 HARD GATE의 본질이다.

#### Checkpoint: 파이프라인 상태 저장 (구현 중)

현재는 Streamlit 세션이 끊기면 생성한 데이터(parquet은 저장됨)는 남지만, "어디까지 했는지"는 사라진다. 벤치마크를 돌렸는지, GAN 최적화 결과가 뭐였는지, 다시 앱을 열면 알 수 없다.

도입하려는 체크포인트:

```json
{
  "session_id": "2025-04-10_14:30",
  "current_stage": "benchmark_done",
  "schema_version": "v2_reviewed",
  "generation": {
    "model_id": "pouch_60Ah_NMC811",
    "n_rows": 10000,
    "parquet_path": "data/generated/ui_pouch_60Ah_NMC811_10000.parquet",
    "quality_gate_passed": true
  },
  "benchmark": {
    "best_method": "correlation",
    "best_score": 0.82,
    "performance_gate_passed": true
  },
  "optimization": null
}
```

JStack의 `CHECKPOINT.md` 상태머신과 같은 역할이다. 세션이 끊겨도 다음에 앱을 열면 "벤치마크까지 완료, GAN 최적화 대기 중" 상태에서 바로 이어서 작업할 수 있다.

## 코드 하네스 vs 데이터 하네스 — 같은 뼈대, 다른 살

| 구성 요소 | JStack (코드 하네스) | Synth-Analyzer (데이터 하네스) |
|---|---|---|
| Rules | CLAUDE.md, NEVER 규칙 | 스키마 (변수 범위, 분포, 단위) |
| Context | CHECKPOINT.md, 프로젝트 문맥 | 논문 기반 물리 모델 (상관관계, 공식) |
| Skills | /office-hours → /plan → /build → /review → /ship | 생성 → 벤치마크 → GPU → GAN → 결과 |
| Evaluation | 테스트 통과 여부 | 난이도별 탐지 점수, 수렴 판정 |
| Human-in-the-loop | 사용자가 /plan 승인 | 도메인 전문가가 스키마 검수 |
| HARD GATE | /review 미통과 → /ship 불가 | (구현 중) 품질 검증 미통과 → 벤치마크 불가 |
| Checkpoint | CHECKPOINT.md 상태머신 | (구현 중) pipeline_state.json |

결국 하네스의 본질은 **"AI의 행동을 구조화하는 것"**이다. 코드를 짜는 AI든, 데이터를 만드는 파이프라인이든, "규칙 → 맥락 → 단계적 실행 → 평가 → 게이트"라는 같은 뼈대 위에서 돌아간다.

## 궁금해진 것: 하네스별 성능 비교

Synth-Analyzer를 만드는 과정에서 현재 **OMC(oh-my-claudecode), GStack, Everything Claude Code** 세 가지 하네스를 모두 내려받아서 혼재해서 쓰고 있다. 각각의 스킬과 에이전트를 섞어 쓰다 보니, 문득 궁금해졌다 — **같은 작업을 시켰을 때 어떤 하네스가 더 잘 하는가?**

예를 들면:

- **코드 리뷰**: 어떤 하네스가 더 꼼꼼하게 버그를 잡는가?
- **디버깅**: 어떤 하네스가 더 빨리 근본 원인을 찾는가?
- **계획 수립**: 어떤 하네스가 더 현실적이고 실행 가능한 플랜을 내놓는가?
- **코드 생성**: 어떤 하네스가 첫 번째 시도에서 작동하는 코드를 만드는가?

1주차에는 G Stack과 OMC를 **설계 관점**에서 비교했다 — 파일 크기, 외부 의존성, 프롬프트 깊이 같은 정적인 지표로. 하지만 실제로 쓰면서 느끼는 건, **설계상 좋아 보이는 것과 실전에서 잘 작동하는 것은 다르다**는 점이다.

하네스를 연구하는 사람으로서, 이건 꼭 해봐야 할 실험이라고 생각한다. 단순히 "어느 게 더 좋다"를 가리는 게 아니라, **어떤 상황에서 어떤 하네스가 강한지** 패턴을 찾는 것이 목표다. 마치 Synth-Analyzer가 "어떤 난이도에서 어떤 분석이 가장 효과적인가"를 찾듯이, 하네스에 대해서도 같은 질문을 던져볼 수 있지 않을까.

## 이번 주 배운 것

### 1. 하네스는 코드 생성에만 해당하는 개념이 아니다

1주차에 "하네스 = AI 에이전트의 OS"라고 정의했는데, 이번 주에 데이터 파이프라인을 만들면서 그 정의가 더 넓어졌다. 하네스는 **"구조화된 자동화 시스템의 운영 체계"**에 더 가깝다.

### 2. 도메인 지식이 하네스의 품질을 결정한다

JStack은 소프트웨어 엔지니어링 도메인 지식(코드 리뷰, 테스트, 배포)을 하네스화한 것이다. Synth-Analyzer는 배터리 제조 공정의 도메인 지식(물리 법칙, 공정 변수, 불량 메커니즘)을 하네스화한 것이다. **하네스의 Rules와 Context가 얼마나 정확하냐가 결과물의 품질을 결정한다.**

### 3. "에러가 나서 못 가는 것"과 "게이트가 막는 것"은 다르다

HARD GATE를 고민하면서 깨달았다. 현재 Synth-Analyzer는 데이터 없이 벤치마크를 누르면 에러가 난다. 하지만 이건 게이트가 아니라 그냥 실패다. 진짜 HARD GATE는 **왜 못 가는지 알려주고, 어떻게 하면 통과할 수 있는지 안내하는 것**이다.

### 4. 하네스를 만드는 것보다 하네스를 쓰는 게 더 많은 걸 가르쳐준다

1주차에는 JStack이라는 하네스를 설계했다. 3주차에는 실무 도구를 만들면서 하네스적 사고를 적용했다. 후자에서 훨씬 더 많은 것을 배웠다. 하네스의 각 요소가 왜 필요한지, 어디에서 빠지면 문제가 생기는지는 **실제로 써봐야 체감된다.**
