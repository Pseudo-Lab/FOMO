# G Stack과 OMC를 해부하고, 나만의 하네스(JStack)를 설계한 이야기

> 하네스 엔지니어링 Deep Dive — OMC vs G Stack 비교 분석부터 4계층 범용 하네스 설계까지

---

## 왜 이걸 하게 됐나

나는 Claude Code로 ProcessLens라는 제조 공정 분석 SaaS를 만들고 있는 솔로 파운더다. 이미 9개의 문서 체계(exec-plans, product-specs, ARCHITECTURE.md, AGENTS.md 등)를 갖추고 있었고, 이것 자체가 "하네스"라는 걸 나중에 알게 됐다.

문제는 이거였다: **ProcessLens 전용 하네스는 있지만, 새로운 프로젝트를 시작할 때마다 처음부터 다시 만들어야 한다.** 위스키 테이스팅 노트 앱을 만들고 싶으면? 주식 자동매매 시스템을 만들고 싶으면? 매번 문서 체계를 새로 짜야 한다.

그래서 **범용 하네스**를 만들기로 했다. 어떤 프로젝트에서든 작동하고, 프로젝트별 도메인 지식은 자동으로 생성해주는 하네스.

그리고 좋은 레퍼런스가 두 개 있었다: **Garry Tan의 G Stack**과 **oh-my-claudecode(OMC)**.

---

## 첫 번째 공부: OMC vs G Stack, 뭐가 다른가

### G Stack — "역할별 모드 전환"
YC CEO Garry Tan이 만든 Claude Code 스킬 모음. `/office-hours`, `/plan-ceo-review`, `/plan-eng-review`, `/review`, `/qa`, `/ship` 같은 슬래시 커맨드로 Claude를 다른 역할로 전환시킨다.

핵심 철학: *"Planning is not review. Review is not shipping. If you blur all of that together, you usually get a mediocre blend of all four."*

### OMC — "자동화된 멀티 에이전트 오케스트레이션"
32개의 전문 에이전트(architect, executor, designer, qa-tester 등)를 tmux 기반으로 병렬 실행한다. 자연어로 지시하면 OMC가 알아서 적절한 에이전트에게 위임한다.

### 근본적 차이: "누가 오케스트레이터인가"

이 차이를 이해하는 데 시간이 걸렸다.

- **G Stack**: 사람이 오케스트레이터. 내가 `/plan`을 호출하고, 결과를 확인하고, `/build`를 호출한다. 매 단계 사이에 내 판단이 개입한다.
- **OMC**: AI가 오케스트레이터. "리팩토링 해줘"라고 한마디 하면, AI Head Agent가 알아서 에이전트들을 스폰하고 결과를 합성한다. 내 판단이 개입할 타이밍이 없다.

솔로 파운더로서 내 도메인 전문성(배터리 제조, 위스키 테이스팅 등)이 개입해야 하는 상황에서는 **G Stack 방식이 맞다**고 결론 내렸다.

## 두 번째 공부: 프롬프트를 실제로 까봤다

G Stack의 `/office-hours` SKILL.md.tmpl을 전문으로 읽었다. 517줄, 24.5KB. 여기서 발견한 핵심 패턴들:

### 1. Forcing Questions 패턴
질문이 그냥 "사용자가 누구예요?"가 아니다. 구조가 있다:
- **Ask**: 구체적이고 도전적인 질문
- **Push until**: 만족스러운 답변의 기준
- **Red flags**: 경고해야 할 나쁜 답변 패턴

예시:
```
Q3: Desperate Specificity
Ask: "이것이 가장 필요한 실제 사람의 이름을 대보세요. 
     직함이 뭐예요? 뭘 하면 승진하고, 뭘 못하면 해고되나요?"
Push until: 이름. 역할. 구체적 결과.
Red flags: "헬스케어 기업." "중소기업." 카테고리에 이메일을 보낼 수는 없다.
```

### 2. Operating Principles
"CEO처럼 생각해"가 아니라, CEO가 어떤 원칙으로 판단하는지를 명시한다:
- 구체성만이 유일한 화폐다
- 관심은 수요가 아니다
- 현상 유지가 진짜 경쟁자다

### 3. Anti-slop 규칙
AI가 흔히 빠지는 "칭찬 슬롭"을 명시적으로 금지한다:
- GOOD: "당신은 '공장'이라고 안 했어요 — '천안의 200인 배터리 공장 QC 팀장'이라고 했죠."
- BAD: "타겟 유저를 특정하는 데 뛰어난 구체성을 보여주셨습니다."

### 4. HARD GATE
도구 수준에서 역할 경계를 강제한다:
```yaml
allowed-tools:
  - Bash, Read, Grep, Glob, Write, Edit, AskUserQuestion
  # 코드 실행 도구가 없음 = 코드를 쓸 수 없음
```

### OMC와의 결정적 차이: 프롬프트 밀도

| 차원 | G Stack /office-hours | OMC /deep-interview |
|------|----------------------|---------------------|
| 프롬프트 길이 | 517줄 / 24.5KB | ~50줄 |
| 질문 구조 | 6개 Forcing Questions (Ask/Push/Red flags) | "소크라테스식 질문" (구조 미명시) |
| Anti-slop | Good/Bad 예시 명시 | 없음 |
| 도구 제한 | allowed-tools 명시 | 없음 |

**결론: 적은 수의 깊은 프롬프트가 많은 수의 얕은 프롬프트보다 효과적이다.**

## 세 번째 공부: "프롬프트가 길면 안 된다"의 진짜 의미

여기서 한동안 헷갈렸다. FOMO 발표에서 "컨텍스트 윈도우 40% 넘으면 Dumb Zone"이라고 했는데, G Stack은 24.5KB짜리 프롬프트를 쓴다. 상충되지 않나?

**아니다.** 핵심은 "언제 로딩되느냐"다.

```
항상 로딩:  CLAUDE.md (~500 토큰)        ← 이게 짧아야 함
호출 시만:  /office-hours (~6,000 토큰)  ← 이건 길어도 됨
```

CLAUDE.md = 항상 메모리에 상주하는 OS 커널 → 작아야 한다.
커맨드 파일 = 필요할 때만 올라오는 앱 → 길어도 된다.

`/office-hours`를 호출하면 office-hours.md가 컨텍스트에 들어오고, `/review`를 호출하면 review.md가 들어온다. 동시에 둘 다 들어오지 않는다.

## 네 번째 공부: 하네스의 본질

여기서도 한참 헤맸다. "스킬을 만들면 그게 하네스 아니야?"

**아니다.** 스킬은 하네스의 일부일 뿐이다.

```
하네스 = CLAUDE.md + Rules + Commands(스킬) + Hooks + Sandbox + Memory
```

아이폰 비유: 카메라 앱이 좋다고 아이폰 = 카메라 앱은 아니다. 하네스는 OS 전체다.

FOMO 발표의 4계층 프레임워크로 정리하면:
```
L4 통제 (Control):    CLAUDE.md + rules/        ← 항상 작동
L3 실행 (Execution):  commands/*.md              ← 호출 시만 로딩  
L2 품질 (Quality):    settings.json + hooks      ← 항상 작동
L1 구조 (Structure):  CHECKPOINT.md + AUDIT.log  ← 항상 작동
```

## 다섯 번째 공부: 에이전트 vs 기어

"에이전트를 몇 명 배분할지"를 먼저 정해야 하지 않나? 라고 생각했는데, 그건 OMC식 사고였다.

G Stack식(우리 하네스)에서는 "에이전트 배분"이 아니라 "모드(기어) 설계"가 맞는 질문이다. 자동차 기어처럼, 같은 Claude 세션이지만 `/plan`일 때와 `/review`일 때 완전히 다른 사고방식으로 작동하게 만드는 거다.

나중에 프로젝트가 커져서 병렬 처리가 필요하면, 그때 OMC식 에이전트를 추가하면 된다. 이게 Rippable 원칙 — 나중에 떼어내거나 붙일 수 있게.

---

## 설계 결정들

### 5단 기어 선택
```
/office-hours → /plan → /build → /review → /ship
```
3단(Plan/Build/Review)이면 `/plan`이 너무 많은 역할을 하고, 4단이면 `/plan`이 여전히 무겁다. 5단으로 역할을 완전히 분리했다.

### 범용 + 프로젝트별 분리
```
~/.claude/CLAUDE.md         ← 범용 (모든 프로젝트 공통)
{project}/CLAUDE.md         ← 프로젝트 전용 (/office-hours가 자동 생성)
```
범용 하네스 = "어떤 도메인이든 올바른 질문을 던지고 구조화하는 하네스".
도메인 지식은 `/office-hours`가 질문을 통해 끌어내서 프로젝트 CLAUDE.md에 저장한다.

### 새 프로젝트 엄격 + 기존 프로젝트 유연
- 새 프로젝트: 반드시 `/office-hours`부터. 단계 건너뛰기 금지.
- 기존 프로젝트: 작업 성격에 맞는 단계에서 시작 가능. 단, 확인 필수.

### 복잡도 게이트
Simple(1-3일) / Medium(1-4주) / Complex(1개월+)로 자동 분기. 위스키 앱(Simple)과 ProcessLens(Complex)에서 다른 깊이의 인터뷰가 적용된다.

### G Stack / OMC와 비교

| 지표 | G Stack | OMC | JStack |
|------|---------|-----|--------|
| 크기 | 150KB+ | 300KB+ | 44KB |
| 외부 의존성 | Bun, Playwright | npm, tmux | 없음 |
| 프롬프트 밀도 | 높음 | 낮음 | 높음 |
| 단계 간 연결 | ls로 파일 탐색 | 세션 메모리 | CHECKPOINT 상태머신 |
| 프로젝트 자동 생성 | ✕ | ✕ | ✅ |
| 아키텍처 보호 | ✕ | △ | ✅ |
| 스코프 크리프 방지 | ✕ | ✕ | ✅ |
| 병렬 실행 | △ | ✅ | ✕ (추후) |
| 모델 라우팅 | ✕ | ✅ | ✕ (추후) |

### JStack만의 차별점 (양쪽 모두에 없는 것)

1. **프로젝트 CLAUDE.md 자동 생성** — /office-hours가 질문 → 디자인 문서 + 프로젝트별 하네스를 자동 생성
2. **CHECKPOINT.md 상태머신** — Phase 필드가 현재 단계를 추적하고, 각 커맨드가 올바른 단계인지 자동 검증
3. **복잡도 게이트** — Simple/Medium/Complex 자동 분기
4. **아키텍처 보호 + 스코프 크리프 방지** — /build에서 설계 변경을 물리적으로 차단하고, 계획에 없는 기능 추가를 금지
5. **제로 의존성** — 마크다운 파일만으로 작동

## 배운 것들

### 1. 프롬프트 밀도 > 프롬프트 수
OMC의 32개 에이전트보다 G Stack의 6개 깊은 스킬이 낫다. 에이전트 수를 늘리는 것보다 각 프롬프트의 지침 밀도를 높이는 게 결과 품질을 결정한다.

### 2. 제약이 품질을 만든다
"하지 마"를 명시하는 게 "해라"를 명시하는 것보다 AI 행동 제어에 효과적이다. HARD GATE, allowed-tools 제한, NEVER 규칙 — 이런 부정형 제약이 프롬프트의 핵심이다.

### 3. 상태 전달은 파일 시스템이 답이다
OMC처럼 세션 메모리에 의존하면 세션이 끊길 때 맥락이 사라진다. CHECKPOINT.md에 상태를 기록하면 다음 세션에서 이어서 작업할 수 있다.

### 4. 범용 하네스 = "좋은 질문의 체계"
모든 도메인을 아는 하네스가 아니라, 어떤 도메인이든 올바른 질문을 던지고 구조화하는 하네스. 도메인 지식은 사람의 머릿속에서 나오고, 하네스는 그걸 끌어내는 역할이다.

### 5. Rippable하게 만들어라
모델이 좋아지면 하네스의 일부는 불필요해진다. 영원한 요새가 아니라, 필요 없는 레이어를 떼어낼 수 있는 모듈식으로 설계해야 한다.
