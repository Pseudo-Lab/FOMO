### 1.2 Nginx (Reverse Proxy)

#### 왜 이런 구조인가?

```
문제: 서비스마다 포트가 다르면 사용자가 혼란스러움
     - LangGraph: 2024
     - Gateway: 8001  
     - Frontend: 3000
     
     "어디로 접속해야 하지...?"

해결: Nginx가 단일 입구 역할
     → localhost:2026 하나만 기억하면 됨
     → 내부적으로 알아서 분배
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **단일 진입점** | 포트 2026 하나로 모든 서비스 접근 |
| 2 | **라우팅** | `/api/langgraph/*` → LangGraph, 나머지 → Gateway |
| 3 | **프론트엔드 서빙** | 웹 UI 정적 파일 제공 |

### 1.3 LangGraph Server + Lead Agent

#### 왜 이런 구조인가?

```
문제: AI 대화 처리는 "상태"가 있음
     - 대화 기록 유지
     - 실행 중인 작업 추적
     - 도구 실행 결과 관리
     
     단순한 REST API로는 부족함

해결: LangGraph = 상태 기반 워크플로우 엔진
     - StateGraph로 대화 흐름 관리
     - 체크포인터로 상태 저장/복구
     - 스트리밍 응답 지원
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **대화 처리** | 사용자 메시지 → AI 응답 생성 (상태 유지) |
| 2 | **도구 실행** | bash, 파일 읽기/쓰기, 웹 검색 등 실제 작업 수행 |
| 3 | **작업 분배** | 복잡한 작업을 Subagent에게 위임 |

### 1.4 Gateway API

#### 왜 이런 구조인가?

```
문제: LangGraph Server에 모든 기능 넣으면?
     - 너무 무거워짐
     - 설정 변경할 때마다 재시작?
     - AI 실행과 관리 기능이 섞임
     
해결: 실무(AI)와 관리(설정)를 분리
     
     LangGraph = 실무팀 (대화, 도구 실행)
     Gateway = 지원팀 (설정, 파일 관리, 메모리)
     
장점:
     - Gateway 여러 개 띄워도 됨 (stateless)
     - Gateway 죽어도 AI 대화는 계속됨
     - 설정 변경이 AI 실행에 영향 안 줌
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **설정 관리** | 모델 목록, MCP 서버, 스킬 켜기/끄기 |
| 2 | **파일 처리** | 업로드 (PDF/PPT 자동 변환), 아티팩트 다운로드 |
| 3 | **메모리 조회** | 저장된 사용자 정보 읽기/갱신 |

### 1.5 미들웨어 체인 (10개)

#### 왜 이런 구조인가?

```
문제: Lead Agent가 모든 걸 직접 처리하면?
     - 코드가 복잡해짐
     - 관심사가 섞임 (대화 + 파일 + 메모리 + 제한...)
     - 기능 추가/제거가 어려움

해결: 미들웨어 패턴 = "비서들이 앞뒤로 서있음"

┌─────────────────────────────────────────────────┐
│  메시지 도착                                    │
│      ↓                                          │
│  [Before 미들웨어들]                            │
│  - ThreadData: "폴더 만들어둘게요"              │
│  - Uploads: "새 파일 있어요"                    │
│  - Sandbox: "실행 환경 준비됐어요"              │
│      ↓                                          │
│  Lead Agent 생각 & 행동                         │
│      ↓                                          │
│  [After 미들웨어들]                             │
│  - Title: "제목 만들어둘게요"                   │
│  - Memory: "기억해둘게요"                       │
│  - SubagentLimit: "3개 넘으면 잘라요"           │
│      ↓                                          │
│  응답 전송                                      │
└─────────────────────────────────────────────────┘

장점:
     - 각 미들웨어가 한 가지 일만 함
     - 순서 변경, 추가/제거 쉬움
     - Lead Agent는 핵심 로직에만 집중
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **상태 준비 (before)** | 스레드 폴더 생성, 샌드박스 획득, 업로드 파일 주입 |
| 2 | **제한 강제 (after)** | Subagent 3개 초과 시 잘라냄, 토큰 한계 시 요약 |
| 3 | **자동화 (after)** | 제목 자동 생성, 메모리 자동 저장 |

| # | 이름 | 역할 | 훅 |
|---|------|------|-----|
| 1 | ThreadDataMiddleware | 스레드별 폴더 생성 | before |
| 2 | UploadsMiddleware | 새 업로드 파일 알림 | before |
| 3 | SandboxMiddleware | 실행 환경 획득 | before |
| 4 | DanglingToolCallMiddleware | 중단된 tool call 정리 | before |
| 5 | SummarizationMiddleware | 토큰 초과 시 요약 | before |
| 6 | TodoListMiddleware | Plan Mode 태스크 관리 | before |
| 7 | TitleMiddleware | 대화 제목 자동 생성 | after |
| 8 | MemoryMiddleware | 대화 기억 저장 | after |
| 9 | ViewImageMiddleware | 이미지 base64 주입 | before |
| 10 | SubagentLimitMiddleware | 최대 3개 강제 | after |
| 11 | ClarificationMiddleware | 명확화 요청 시 중단 | after |

### 1.6 Subagent 시스템

#### 왜 이런 구조인가?

```
문제: 복잡한 작업을 Lead Agent 혼자 다 하면?
     
     "2026년 AI 트렌드 리포트 만들어줘"
     
     → 웹 리서치 (5분)
     → 논문 분석 (7분)  
     → 경쟁사 조사 (4분)
     → 문서 작성 (3분)
     
     순차 실행 = 19분
     컨텍스트 윈도우 폭발
     실패하면 처음부터 다시

해결: 팀장(Lead) + 팀원(Subagent) 구조

     Lead Agent: "이 작업 3개로 나눠서 병렬로 해"
     
     ┌─────────┐ ┌─────────┐ ┌─────────┐
     │웹 리서치│ │논문 분석│ │경쟁사조사│  ← 병렬!
     └────┬────┘ └────┬────┘ └────┬────┘
          └──────────┴──────────┘
                     ↓
     Lead Agent: 결과 종합 → 리포트 생성
     
     병렬 실행 = 7분 (가장 긴 작업 기준)
     각 Subagent는 독립 컨텍스트 (섞이지 않음)
     하나 실패해도 다른 건 계속 진행
```

#### 왜 최대 3개? 왜 15분 타임아웃?

```
최대 3개 제한:
- 너무 많으면 리소스 고갈
- 관리가 어려워짐
- 대부분의 작업은 3개면 충분

15분 타임아웃:
- 무한 실행 방지
- 리소스 회수
- 문제 있는 Subagent 정리
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **작업 분해** | 큰 작업을 독립적인 작은 작업으로 나눔 |
| 2 | **병렬 실행** | 최대 3개 동시 실행 (dual thread pool) |
| 3 | **컨텍스트 격리** | 각 Subagent는 독립 컨텍스트, 서로 간섭 안 함 |

### 1.7 Sandbox 시스템

#### 왜 이런 구조인가?

```
문제 1: AI가 실제 시스템 경로를 알면 위험함
     
     AI: "rm -rf /home/user/important_files" 
     → 😱

문제 2: 스레드마다 파일이 섞이면 안 됨
     
     스레드 A의 output.txt vs 스레드 B의 output.txt
     → 덮어쓰기? 혼란?

해결: 가상 경로 시스템

     AI가 보는 것              실제 위치
     ──────────────────────────────────────
     /mnt/user-data/workspace → .deer-flow/threads/abc123/...
     /mnt/user-data/uploads   → .deer-flow/threads/abc123/...
     /mnt/user-data/outputs   → .deer-flow/threads/abc123/...
     
장점:
     - AI는 /mnt/user-data/만 알면 됨 (일관성)
     - 실제 시스템 경로 노출 안 됨 (보안)
     - 스레드별로 완전히 격리 (안전)
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **경로 격리** | 가상 경로 ↔ 실제 경로 자동 변환 |
| 2 | **안전한 실행** | Docker 컨테이너 또는 로컬 격리 환경 |
| 3 | **파일 도구 제공** | bash, ls, read_file, write_file, str_replace |

### 1.8 Memory 시스템

#### 왜 이런 구조인가?

```
문제: 대부분의 AI는 대화가 끝나면 다 잊어버림

     [어제 대화]
     사용자: "나 AICESS에서 TTS 엔지니어로 일해"
     AI: "네, 알겠습니다!"
     
     [오늘 대화]
     사용자: "저번에 얘기한 TTS 프로젝트 이어서..."
     AI: "무슨 TTS요? 어디서 일하시는데요?"
     → 😤

해결: Memory 시스템 = 장기 기억

     ┌─────────────────────────────────────────┐
     │  대화 끝                                │
     │      ↓                                  │
     │  MemoryMiddleware가 대화 수집           │
     │      ↓                                  │
     │  30초 대기 (debounce)                   │
     │      ↓                                  │
     │  LLM이 사실 추출                        │
     │  "민정님은 AICESS TTS 엔지니어"         │
     │  "aivox-tts 프로젝트 진행 중"           │
     │      ↓                                  │
     │  memory.json에 저장                     │
     │      ↓                                  │
     │  [다음 대화]                            │
     │  system prompt에 주입                   │
     │  "<memory>민정님은 TTS...</memory>"     │
     └─────────────────────────────────────────┘
```

#### 왜 30초 debounce?

```
문제: 매 메시지마다 LLM 호출하면?
     - 비용 폭탄
     - 불필요한 API 호출

해결: 30초 모았다가 한 번에 처리
     - 대화가 끝날 때까지 기다림
     - 배치로 효율적 처리
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **사실 추출** | 대화에서 중요 정보 자동 추출 (LLM 사용) |
| 2 | **영구 저장** | `memory.json`에 facts + context 저장 |
| 3 | **컨텍스트 주입** | 다음 대화 시 system prompt에 top 15 facts 삽입 |

### 1.9 Skill 시스템

#### 왜 이런 구조인가?

```
문제: 모든 역량을 하드코딩하면?
     - 코드 수정 필요
     - 배포 필요
     - 사용자가 커스터마이징 못함

해결: Skill = Markdown 파일로 역량 정의

     skills/
     ├── public/           # 기본 제공
     │   ├── research/SKILL.md
     │   ├── report-generation/SKILL.md
     │   └── slide-creation/SKILL.md
     └── custom/           # 사용자 정의
         └── tts-evaluation/SKILL.md
         
장점:
     - 코드 수정 없이 역량 추가
     - 사용자가 직접 만들 수 있음
     - 켜기/끄기 가능
```

#### 왜 Progressive Loading?

```
문제: 모든 스킬을 항상 로드하면?
     - 컨텍스트 윈도우 낭비
     - 토큰 비용 증가
     - 불필요한 정보로 AI 혼란

해결: 필요한 스킬만 로드

     "슬라이드 만들어줘" 
     → slide-creation/SKILL.md만 로드
     
     "리서치 해줘"
     → research/SKILL.md만 로드
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **역량 정의** | Markdown 파일로 워크플로우/베스트 프랙티스 정의 |
| 2 | **Progressive Loading** | 필요한 스킬만 로드 (컨텍스트 절약) |
| 3 | **확장성** | `skills/custom/`에 사용자 정의 스킬 추가 가능 |

## Part 2: SWE-Agent 아키텍처

### 2.1 ACI (Agent-Computer Interface) — 전체 개념

#### 왜 이런 구조인가?

```
핵심 통찰: 기존 인터페이스는 "인간"을 위해 설계됨

     GUI (VSCode, IDE) → 인간의 시각 + 마우스/키보드
     CLI (bash, shell) → 인간의 타이핑 + 화면 읽기
     
     LLM은 "새로운 종류의 사용자"
     - 컨텍스트 윈도우 제한
     - 패턴 매칭 기반 추론
     - 너무 많은 정보에 혼란
     - 구조화된 피드백에 효과적

문제: LLM에게 그냥 Linux shell 주면?

     $ cat huge_file.py
     → 10,000줄 전체 출력 (컨텍스트 폭발)
     
     $ grep -r "error" .
     → 2,847개 결과 (노이즈 과다)
     
     $ vim file.py (편집 후)
     → 뭐가 바뀌었는지 모름

해결: LLM 전용 인터페이스 (ACI)

     파일 보기: 100줄 윈도우 (컨텍스트 절약)
     검색: 50개 제한 (노이즈 제거)
     편집: 라인 범위 + 변경 내용 표시
     가드레일: 구문 오류 시 편집 거부
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **추상화** | 복잡한 bash 명령을 단순한 전용 명령으로 대체 |
| 2 | **의도적 제한** | 출력량, 검색 결과 수를 제한 (노이즈 제거) |
| 3 | **구조화된 피드백** | 모든 응답을 일관된 템플릿으로 제공 |

### 2.2 File Viewer (파일 뷰어)

#### 왜 이런 구조인가?

```
문제: cat으로 파일 전체 보여주면?

     $ cat huge_file.py
     
     [10,000줄 전체 출력]
     
     → 컨텍스트 윈도우 낭비
     → 정작 필요한 부분 찾기 어려움
     → 토큰 비용 폭발

해결: 윈도우 기반 뷰어 (100줄씩)

     [File: /repo/src/models.py (2,500 lines)]
     Line 401:     def process(self, data):
     Line 402:         """Process the input data."""
     Line 403:         if not data:
     Line 404:             raise ValueError("Empty data")
     ...
     Line 500:     def cleanup(self):
     [Showing lines 401-500 of 2,500]
     
장점:
     - 필요한 부분만 집중
     - 라인 번호로 정확한 위치 파악
     - scroll/goto/search로 탐색
```

#### 왜 100줄?

```
하이퍼파라미터 스윕 결과:
- 50줄: 너무 적음, 맥락 파악 어려움
- 100줄: 최적
- 200줄: 컨텍스트 낭비 시작
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **윈도우 제한 (100줄)** | 컨텍스트 낭비 방지, 필요한 부분만 집중 |
| 2 | **라인 번호 표시** | edit 명령에서 정확한 위치 지정 가능 |
| 3 | **탐색 명령** | `goto`, `scroll_up`, `scroll_down`, `search_file` |

### 2.3 Search Commands (검색 명령)

#### 왜 이런 구조인가?

```
문제: grep으로 검색하면?

     $ grep -r "error" .
     
     src/models.py:142:    raise ValueError("error")
     src/models.py:156:    log.error("error occurred")
     src/utils.py:23:      # handle error
     ... (2,844개 더)
     
     → 컨텍스트 오염
     → LLM이 어디서 시작해야 할지 모름
     → 노이즈에 파묻힘

해결: 결과 50개 제한 + 쿼리 정제 유도

     search_dir "ValueError"
     
     Found 23 matches:
     src/models.py:142
     src/models.py:156
     ...
     
     ─────────────────────────
     
     search_dir "error"
     
     Search returned 847 results.
     Showing first 50.
     Please refine your query for more specific results.
     
     → 에이전트: "더 구체적으로 검색해야겠다"
     → search_dir "ValueError in models"
```

#### 왜 50개 제한이 성능을 높이는가?

```
직관: 더 많은 정보 = 더 좋은 결과?
현실: LLM은 너무 많은 정보에 혼란

제한의 효과:
1. 에이전트가 더 구체적인 쿼리 작성
2. 노이즈 제거 → 신호에 집중
3. 컨텍스트 윈도우 효율적 사용
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **결과 50개 제한** | 초과 시 "Please refine your query" 메시지 |
| 2 | **3가지 검색 범위** | `find_file`(파일명), `search_dir`(디렉토리), `search_file`(파일 내) |
| 3 | **요약된 출력** | 파일 경로, 라인 번호, 매칭 스니펫만 표시 |

---

### 2.4 Edit Command (편집 명령)

#### 왜 이런 구조인가?

```
문제: 기존 편집 방식

     1. 파일 전체 읽기 (cat)
     2. 수정할 부분 찾기
     3. 전체 파일 다시 쓰기
     4. 변경 확인을 위해 다시 읽기
     
     → 4단계, 비효율적
     → 뭐가 바뀌었는지 불명확

해결: 라인 범위 기반 편집 + 자동 피드백

     edit 404:407
         if data is None:
             raise ValueError("Data cannot be None")
         return self._transform(data)
     end_of_edit
     
     [File: models.py]
     [Lines 404-407 replaced]
     [Showing updated content:]
     Line 403:     def process(self, data):
     Line 404:         if data is None:
     Line 405:             raise ValueError("Data cannot be None")
     Line 406:         return self._transform(data)
     Line 407:     
     
     → 1단계로 끝
     → 변경 내용 즉시 확인
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **라인 범위 지정** | 정확한 위치에 수정 (404:407 = 404~407번 줄) |
| 2 | **변경 내용 즉시 표시** | 편집 후 자동으로 before/after 보여줌 |
| 3 | **File Viewer 연동** | 편집 후 해당 위치로 뷰어 자동 이동 |

---

### 2.5 Linter Guard (린터 가드레일)

#### 왜 이런 구조인가?

```
문제: 잘못된 코드가 저장되면?

     edit 404:407
         if data is None    ← 콜론 누락!
             raise ValueError("...")
     end_of_edit
     
     [저장됨]
     
     → 이후 모든 작업이 구문 오류 위에서 진행
     → 테스트 실패, 원인 파악 어려움
     → "좌초(stranding)" 상태

해결: 편집 전 린터 검사 → 오류 시 거부

     edit 404:407
         if data is None    ← 콜론 누락
             raise ValueError("...")
     end_of_edit
     
     [Edit REJECTED]
     SyntaxError at line 404: expected ':'
     
     Before: if not data:
     After:  if data is None
                            ^ missing ':'
     
     Please fix and try again.
     
     → 잘못된 코드가 저장 자체가 안 됨
     → 즉각적 피드백
     → 복구 비용 없음
```

#### Ablation 결과: 가장 큰 기여

```
SWE-bench Lite (300개 태스크)

Baseline (shell만):     7.3%
+ File Viewer:         10.0%  (+2.7pp)
+ Search:              12.7%  (+2.7pp)
+ Edit:                14.7%  (+2.0pp)
+ Linter Guard:        18.0%  (+3.3pp) ← 가장 큰 단일 기여!
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **구문 오류 시 edit 거부** | 잘못된 코드가 저장되는 것 자체를 막음 |
| 2 | **에러 위치 + 메시지 제공** | 어디가 틀렸는지 정확히 알려줌 |
| 3 | **before/after 스니펫** | 뭐가 잘못됐는지 시각적으로 비교 |

### 2.6 Context Management (컨텍스트 관리)

#### 왜 이런 구조인가?

```
문제: 에이전트가 오래 실행되면?

     스텝 1: 파일 열기
     스텝 2: 검색
     스텝 3: 편집 (실패)
     스텝 4: 재시도 (실패)
     스텝 5: 재시도 (성공)
     ...
     스텝 20: ???
     
     → 과거 에러 메시지가 컨텍스트 차지
     → 중요한 정보가 밀려남
     → 컨텍스트 윈도우 낭비

해결: 히스토리 압축 + 일관된 템플릿

     히스토리 처리:
     - 성공한 명령만 유지
     - 과거 에러 메시지 생략 (최초 1개만)
     - 출력 없으면 "Your command ran successfully"
     
     일관된 템플릿:
     ┌─────────────────────────────────────┐
     │ (Open file: /repo/src/models.py)    │
     │ [File content]                       │
     │ (Showing lines 401-500 of 2,500)    │
     └─────────────────────────────────────┘
     
     → 모든 명령 결과가 같은 형식
     → 에이전트가 파싱하기 쉬움
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **히스토리 압축** | 과거 에러 메시지 생략, 중요한 것만 유지 |
| 2 | **일관된 출력 템플릿** | 모든 명령 결과를 같은 형식으로 |
| 3 | **"no output" 처리** | 출력 없으면 "Your command ran successfully" |

### 2.7 ReAct Loop (실행 루프)

#### 왜 이런 구조인가?

```
문제: LLM이 바로 명령만 출력하면?

     search_dir "bug"
     edit 142:145
     ...
     
     → 왜 그 명령을 실행하는지 모름
     → 디버깅 어려움
     → 계획 없이 즉흥적

해결: Thought + Command 강제

     Thought: 이슈에서 "bug_function"이 언급됐으니 
              먼저 해당 함수를 찾아야 한다.
     Command: search_dir "bug_function"
     
     Thought: 검색 결과 models.py 142번 줄에 있다.
              해당 파일을 열어서 확인하자.
     Command: open models.py 142
     
     → 추론 과정이 명시됨
     → 디버깅 가능
     → 계획적 실행
```

#### 왜 단일 명령?

```
문제: 여러 명령 한 번에 실행하면?

     Command: search_dir "bug" && open models.py && edit 142:145
     
     → 첫 번째가 실패하면 나머지도 영향
     → 어디서 문제인지 파악 어려움
     → 복잡도 증가

해결: 매 스텝에 하나의 명령만

     → 결과 확인 후 다음 결정
     → 실패해도 바로 대응 가능
     → 단순하고 예측 가능
```

#### 핵심 기능 3가지

| # | 기능 | 설명 |
|---|------|------|
| 1 | **Thought 강제** | 명령 전에 반드시 이유/계획 작성 |
| 2 | **단일 명령** | 매 스텝에 하나의 명령만 실행 (복잡도 제어) |
| 3 | **Malformed 처리** | 잘못된 형식이면 재시도 요청 |

## Part 3: DeerFlow vs SWE-Agent 비교

### 3.1 설계 철학 대비

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   DeerFlow                              SWE-Agent               │
│   "환경을 완전히 통제"                  "인터페이스를 정밀 설계" │
│                                                                 │
│   • 8개 구조로 전체 환경 구성           • 7개 ACI 컴포넌트       │
│   • Subagent로 작업 분산                • 단일 에이전트 집중     │
│   • Memory로 장기 기억                  • 가드레일로 실수 방지   │
│   • Middleware로 제어                   • 제한으로 노이즈 제거   │
│                                                                 │
│   "많은 손, 하나의 하네스"              "하나의 손, 최적의 도구" │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 구조 대비

| DeerFlow | SWE-Agent | 비고 |
|----------|-----------|------|
| Nginx (라우팅) | — | DeerFlow만 |
| LangGraph (AI 실행) | ReAct Loop | 유사 |
| Gateway (설정 관리) | — | DeerFlow만 |
| 미들웨어 10개 | Linter + Context Mgmt | 유사한 역할 |
| Subagent (분산) | — | DeerFlow만 |
| Sandbox (격리) | Docker 컨테이너 | 유사 |
| Memory (기억) | — | DeerFlow만 |
| Skill (확장) | — | DeerFlow만 |
| — | File Viewer | SWE-Agent만 |
| — | Search Commands | SWE-Agent만 |
| — | Edit Command | SWE-Agent만 |

### 3.3 핵심 차이

| 측면 | DeerFlow | SWE-Agent |
|------|----------|-----------|
| **문제 해결 방식** | 인프라 구축 | 인터페이스 최적화 |
| **복잡도** | 높음 (8개 구조, 10개 미들웨어) | 낮음 (7개 ACI 컴포넌트) |
| **에이전트 수** | Lead + 최대 3 Subagents | 단일 에이전트 |
| **메모리** | 영구 저장 (세션 간 유지) | 없음 |
| **가드레일** | 미들웨어로 제어 | 린터 + 검색 제한 |
| **적합한 작업** | 장시간 복잡한 워크플로우 | 단일 이슈 해결 |

## Part 4: 핵심 인사이트

### 4.1 인터페이스가 모델보다 중요할 수 있다

```
SWE-Agent 논문 핵심 발견:

같은 GPT-4 Turbo, 같은 SWE-bench 태스크

Baseline (Linux shell): 7.3%
SWE-Agent (ACI):       18.0%
─────────────────────────────
차이: +10.7pp (147% 상대 향상)

변수: 인터페이스만

→ 모델 바꾸기 전에 인터페이스를 최적화하라
```

### 4.2 제한이 성능을 높인다

```
직관에 반하는 발견:

| 제한               | 효과                              |
|--------------------|-----------------------------------|
| 검색 50개 제한     | 더 구체적인 쿼리 → 정확도 향상    |
| 파일 100줄 제한    | 필요한 부분 집중 → 효율 향상      |
| 린터 거부          | 잘못된 상태 방지 → 복구 비용 제거 |
| Subagent 3개 제한  | 리소스 관리 → 안정적 실행         |

원리: LLM은 너무 많은 정보에 혼란
     노이즈 제거 > 정보 제공
```

### 4.3 가드레일의 ROI가 가장 높다

```
SWE-Agent ablation:
- Linter Guard가 +3.3pp로 가장 큰 기여

DeerFlow:
- SubagentLimitMiddleware: 리소스 고갈 방지
- DanglingToolCallMiddleware: 불완전 상태 방지

교훈: 에이전트가 실수 못하게 막는 것 > 더 나은 도구 제공
```
