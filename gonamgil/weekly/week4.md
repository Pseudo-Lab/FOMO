# FOMO Crew W4: Agent-Agnostic — 어떤 에이전트가 와도 같은 하네스

> **"하네스는 말을 가리지 않는다. 어떤 말이든 마구를 채우면 같은 길을 달린다."**

---

## 1. W1 → W2 → W3 → W4: 지금까지의 여정

| 주차 | 질문 | 답 | 하네스 측면 |
|:---:|:---|:---|:---:|
| **W1** | 왜 구조가 무너지나? | 프레임워크가 하네스 | 제약 |
| **W2** | AI가 도구를 못 고른다? | Skills + Composite | 유도 |
| **W3** | 실패하면 어쩌나? | Self-Correction Loop | 복원 |
| **W4** | **이걸 누가 쓰나?** | **어떤 에이전트든** | **연결** |

W3까지 만든 하네스 시스템 — Guard, Transaction, ATE, Self-Correction Loop — 이 모든 것은 **누군가가 돌려야** 의미가 있다. 그 "누군가"가 Claude Code일 수도, Cursor일 수도, Copilot일 수도 있다.

여기서 선택지가 갈린다:

- **A. 특정 에이전트 전용으로 만든다** → 빠르지만, 에이전트가 바뀌면 다시 만들어야 한다
- **B. 어떤 에이전트든 연결되게 만든다** → 느리지만, 하네스가 에이전트보다 오래 산다

Mandu는 **B**를 선택했다. 이유는 단순하다: **에이전트는 빠르게 바뀌지만, 아키텍처 규칙은 바뀌지 않는다.** 6개월 전 최고의 에이전트가 지금은 구식이 될 수 있다. 하지만 "infra 계층에서만 DB에 접근한다"는 규칙은 에이전트가 바뀌어도 유효하다.

**하네스는 에이전트보다 오래 살아야 한다.**

---

## 2. MCP — 에이전트와 프레임워크 사이의 공용어

어떤 에이전트든 연결하려면 **공통 프로토콜**이 필요하다. Mandu는 **MCP(Model Context Protocol)**를 그 프로토콜로 선택했다.

### MCP란?

MCP는 AI 에이전트와 외부 시스템이 대화하는 표준 프로토콜이다. 핵심은 세 가지:

| 기능 | 설명 | Mandu에서의 역할 |
|:---:|:---|:---|
| **Tools** | 에이전트가 호출할 수 있는 함수 | 85개 MCP 도구 (guard.check, route.add 등) |
| **Resources** | 에이전트가 읽을 수 있는 데이터 | 설정, 라우트 맵, 에러 로그, 스킬 문서 |
| **Prompts** | 에이전트가 참조할 수 있는 템플릿 | new-feature, debug, add-crud 가이드 |

Tools만 있으면 "실행"만 가능하다. Resources가 있어야 에이전트가 현재 상태를 "읽을" 수 있고, Prompts가 있어야 워크플로우의 "맥락"을 이해한다.

### 왜 MCP인가?

```
REST API?    → 에이전트마다 HTTP 클라이언트 구현 필요
gRPC?        → 스키마 컴파일 필요, 에이전트가 지원해야 함
Custom SDK?  → 에이전트마다 SDK 개발 필요

MCP?         → 에이전트가 MCP 클라이언트만 있으면 끝
```

Claude Code, Cursor, Windsurf, VS Code Copilot — MCP를 지원하는 에이전트가 급속히 늘고 있다. Mandu가 MCP 서버로 존재하면, 새 에이전트가 나올 때마다 별도 작업 없이 연결된다.

## 3. Mandu MCP 서버 — 연결의 실체

### stdio 기반 통신

Mandu의 MCP 서버는 **stdio(표준 입출력) 전송**을 사용한다.

```
에이전트 (Claude Code, Cursor, ...)
    ↕ stdin/stdout (JSON-RPC)
Mandu MCP 서버 (bun 프로세스)
    ↕
프로젝트 파일시스템
```

에이전트가 Mandu를 자식 프로세스로 실행하고, stdin/stdout으로 JSON-RPC 메시지를 주고받는다. HTTP 서버도 아니고, 소켓도 아니다. 가장 단순한 프로세스 간 통신이다.

**설정 한 줄이면 연결된다:**

```json
// .mcp.json
{
  "mcpServers": {
    "mandu": {
      "command": "bun",
      "args": ["run", "packages/mcp/src/index.ts"],
      "cwd": "."
    }
  }
}
```

Claude Code든 Cursor든, 이 설정을 읽고 `bun run packages/mcp/src/index.ts`를 실행하면 Mandu 하네스에 연결된다. **에이전트별 어댑터가 없다.** 프로토콜이 같으니 어댑터가 필요 없다.

### Plugin Registry — 도구는 플러그인이다

85개 도구는 중앙 레지스트리(`mcp-tool-registry.ts`)에 플러그인으로 등록된다.

```
registerBuiltinTools()
  ├── spec tools (route.add, validate, ...)
  ├── guard tools (check, negotiate, heal, ...)
  ├── transaction tools (begin, commit, rollback, ...)
  ├── ate tools (auto_pipeline, heal, ...)
  ├── composite tools (feature.create, diagnose, ...)
  └── ... 17개 카테고리
```

각 도구는 동일한 형식을 따른다:

```typescript
{
  name: "mandu.guard.check",        // 이름
  description: "아키텍처 위반 검증",    // 설명 (에이전트가 읽음)
  inputSchema: { ... },              // 입력 스키마 (에이전트가 파라미터 구성)
  annotations: {
    relatedSkills: ["mandu-guard-guide"],  // 관련 스킬 힌트
    isError: false                          // 에러 도구 여부
  },
  execute: async (args) => result    // 실행 함수
}
```

에이전트는 도구의 `name`, `description`, `inputSchema`만 보고 호출을 결정한다. 실행 로직은 Mandu 서버 내부에서 돌아간다. **에이전트는 "무엇을 할 수 있는지"만 알면 되고, "어떻게 하는지"는 몰라도 된다.**

## 4. Resources — 에이전트에게 "눈"을 주다

Tools가 "손"이라면, Resources는 **"눈"**이다. 에이전트가 현재 프로젝트 상태를 읽을 수 있게 해준다.

| Resource URI | 제공하는 정보 |
|:---|:---|
| `mandu://config` | 프레임워크 설정 (Guard 규칙, 프로젝트 구조) |
| `mandu://routes` | 전체 라우트 매니페스트 |
| `mandu://errors` | 현재 Guard 위반 목록 |
| `mandu://slots/{routeId}` | 특정 라우트의 Slot 상태 |
| `mandu://watch/warnings` | Watcher가 감지한 실시간 경고 |

에이전트가 "로그인 페이지를 수정해줘"라고 요청받으면:

1. `mandu://routes`를 읽어서 로그인 라우트를 찾고
2. `mandu://slots/login`을 읽어서 현재 Slot 상태를 확인하고
3. `mandu://errors`를 읽어서 기존 위반이 있는지 체크한 뒤
4. 작업을 시작한다

**Resources 없이 Tools만 있으면**, 에이전트는 "눈 감고 손만 움직이는" 상태다. 현재 상태를 모르고 도구를 호출하니 엉뚱한 결과가 나온다.

---

## 5. Prompts — 에이전트에게 "레시피"를 주다

W2에서 소개한 Skills는 Claude Code 전용이었다. 하지만 MCP Prompts는 **모든 에이전트가 접근할 수 있는 레시피**다.

| Prompt | 역할 |
|:---|:---|
| `new-feature` | 기능 생성 워크플로우 가이드 |
| `debug` | 문제 진단 파이프라인 |
| `add-crud` | CRUD API 템플릿 |

에이전트가 Prompt를 요청하면, Mandu가 현재 프로젝트 상태에 맞는 가이드를 반환한다. "이 프로젝트에서 기능을 추가하려면 이 순서로, 이 도구들을 사용하세요."

```
Skills  → Claude Code 전용 (마크다운 파일)
Prompts → 모든 MCP 에이전트 공용 (프로토콜 레벨)
```

Skills는 더 상세하고 풍부하지만 특정 에이전트에 종속된다. Prompts는 간결하지만 에이전트를 가리지 않는다. **두 레이어가 공존하는 것이 Mandu의 전략이다.**

## 6. Profile — 에이전트 능력에 맞게 도구를 조절하다

에이전트마다 능력이 다르다. Claude Code는 복잡한 멀티스텝을 잘 처리하지만, 간단한 자동완성 에이전트는 단순한 도구만 소화할 수 있다. 85개 도구를 전부 노출하면 가벼운 에이전트가 과부하에 걸린다.

Mandu의 **Profile 시스템**이 이를 해결한다.

```
MANDU_MCP_PROFILE=minimal   → 핵심 도구만 (route.add, guard.check, generate)
MANDU_MCP_PROFILE=standard  → 주요 도구 + composite
MANDU_MCP_PROFILE=full      → 85개 전체
```

| Profile | 도구 수 | 대상 에이전트 |
|:---:|:---:|:---|
| **minimal** | ~15개 | 자동완성, 간단한 코드 생성 에이전트 |
| **standard** | ~40개 | 일반적인 코딩 에이전트 |
| **full** | 85개 | Claude Code 같은 고급 에이전트 |

**같은 하네스, 다른 인터페이스.** Guard 규칙은 동일하다. Transaction도 동일하다. 에이전트가 보는 도구 목록만 다르다. 가벼운 에이전트도 하네스 안에서 안전하게 작업하고, 강력한 에이전트는 하네스의 모든 기능을 활용한다.

## 7. Hook — 에이전트의 행동을 프레임워크가 감시하다

MCP 도구 호출 전후에 **Hook**을 걸 수 있다.

```json
// .claude/settings.json (또는 에이전트별 설정)
{
  "hooks": {
    "PreToolUse": "bunx mandu guard check-file \"$FILE_PATH\"",
    "PostToolUse": "bunx mandu notify-change \"$TOOL_NAME\""
  }
}
```

| 시점 | 용도 |
|:---|:---|
| **PreToolUse** | 도구 실행 전 Guard 검증. 위반 시 실행 차단 |
| **PostToolUse** | 도구 실행 후 변경 기록, 알림, 로깅 |

에이전트가 `route.add`를 호출하기 전에 Guard가 한 번 더 체크하고, 호출 후에 변경 사항을 기록한다. **에이전트가 MCP 도구를 직접 호출해도, Hook이 하네스의 규칙을 강제한다.**

이건 W1의 "프레임워크의 강제력"이 에이전트 연동 레벨에서도 작동한다는 뜻이다. 에이전트가 Guard를 무시하고 직접 파일을 수정하려 해도, Hook이 그 행위를 잡아낸다.

## 8. 전체 그림: Agent-Agnostic 하네스 아키텍처

```
┌──────────────────────────────────────────────────────┐
│  에이전트 계층 (교체 가능)                              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐              │
│  │Claude Code│ │  Cursor  │ │ Copilot  │  ...         │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘              │
│       │             │            │                     │
│       └─────────────┼────────────┘                     │
│                     │ MCP (stdio, JSON-RPC)            │
├─────────────────────┼────────────────────────────────┤
│  하네스 계층 (불변)   │                                 │
│                     ▼                                  │
│  ┌─────────────────────────────────┐                   │
│  │  Mandu MCP Server               │                   │
│  │  ┌───────────┐ ┌──────────────┐│                   │
│  │  │ Profile   │ │    Hooks     ││  ← 진입 제어      │
│  │  └───────────┘ └──────────────┘│                   │
│  │  ┌───────────┐ ┌──────────────┐│                   │
│  │  │  Tools    │ │  Resources   ││  ← 실행 + 관찰   │
│  │  │  (85개)   │ │  (5 URI)     ││                   │
│  │  └───────────┘ └──────────────┘│                   │
│  │  ┌───────────┐ ┌──────────────┐│                   │
│  │  │  Prompts  │ │   Skills     ││  ← 가이드        │
│  │  └───────────┘ └──────────────┘│                   │
│  └─────────────────────────────────┘                   │
│                     │                                  │
│  ┌─────────────────────────────────┐                   │
│  │  Core Systems                    │                   │
│  │  Guard │ Transaction │ ATE │ Brain │               │
│  └─────────────────────────────────┘                   │
│                     │                                  │
├─────────────────────┼────────────────────────────────┤
│  프로젝트 계층       │                                 │
│                     ▼                                  │
│  app/ │ spec/ │ .mandu/ │ routes.manifest.json         │
└──────────────────────────────────────────────────────┘
```

**위(에이전트)는 교체 가능하다. 아래(하네스)는 불변이다.**

어떤 에이전트가 연결되든:
- **같은 Guard 규칙**이 적용된다
- **같은 Transaction**으로 안전망이 보장된다
- **같은 ATE**가 테스트를 생성하고 치유한다
- **같은 Self-Correction Loop**가 돌아간다

에이전트는 "운전자"이고, 하네스는 "도로 + 신호등 + 가드레일"이다. 운전자가 바뀌어도 도로는 같다.

## 9. 이게 왜 중요한가: 에이전트 시대의 생존 전략

AI 에이전트 생태계는 빠르게 변한다. 6개월 전에는 Copilot이 지배적이었고, 지금은 Claude Code와 Cursor가 떠오르고 있다. 6개월 후에는 또 다른 에이전트가 나올 수 있다.

만약 하네스가 특정 에이전트에 종속되어 있다면?

```
에이전트 A 전용 하네스 구축 (3개월)
  → 에이전트 B가 더 좋아짐
  → 하네스 마이그레이션 (2개월)
  → 에이전트 C가 등장
  → 또 마이그레이션...
```

Agent-agnostic 하네스라면?

```
MCP 기반 하네스 구축 (3개월)
  → 에이전트 B 등장 → MCP 연결 (설정 한 줄)
  → 에이전트 C 등장 → MCP 연결 (설정 한 줄)
  → 하네스는 그대로, 에이전트만 교체
```

**하네스에 투자한 시간이 에이전트가 바뀌어도 살아남는다.** 이것이 agent-agnostic 설계의 핵심 가치다.

## 10. 남은 과제: 완벽하지 않다

솔직히, 아직 완벽하지는 않다.

| 과제 | 현재 상태 | 이상적인 상태 |
|:---|:---|:---|
| **Skills 이식성** | Claude Code 전용 마크다운 | MCP Prompts로 통합 |
| **Hook 표준화** | 에이전트마다 Hook 설정 방식이 다름 | MCP 레벨 표준 Hook |
| **Profile 자동 감지** | 수동으로 환경변수 설정 | 에이전트 능력 자동 감지 |
| **실시간 피드백** | Watcher 경고를 에이전트가 폴링 | MCP notification 푸시 |

특히 **Skills의 이식성**이 가장 큰 숙제다. 현재 9개의 상세한 Skills는 Claude Code 전용이고, MCP Prompts는 3개뿐이다. Skills의 풍부함을 MCP Prompts로 옮기는 작업이 필요하다.

하지만 방향은 명확하다: **에이전트 종속 레이어를 줄이고, 프로토콜 레이어를 넓히는 것.**

- **MCP가 "에이전트의 USB-C"가 될 수 있을까요? 아니면 또 다른 표준 전쟁이 올까요?**

## 12. Retrospective

> "W1에서 울타리를 쳤다. W2에서 길을 닦았다. W3에서 안전망을 깔았다.
> W4에서 깨달은 것: 이 모든 걸 '누가 쓰느냐'는 중요하지 않다.
> 중요한 건 '누가 쓰든 같은 규칙이 적용되느냐'다.
> 에이전트는 빠르게 바뀐다. 하네스는 느리게 바뀌어야 한다.
> 변하는 것에 투자하면 변할 때마다 잃는다.
> 변하지 않는 것에 투자하면, 변해도 남는다.
> Mandu의 MCP 서버는 그 '변하지 않는 것'을 향한 시도다."
