# FOMO Crew W1: 웹 프레임워크로 하네스 엔지니어링 도전하기

> **"AI에게 규칙을 말하지 마라. 규칙이 되는 구조를 만들어라."**

---

## 1. 문제: AI와 웹 개발을 하면 할수록 구조가 무너진다

여러 웹 프레임워크를 써가며 다양한 아키텍처로 웹 개발을 해왔다. 특히 Codex-GPT-4.1, Sonnet 3.5 시절부터 AI와 함께 웹 개발을 해왔는데, 기능을 추가할수록 코드가 망가지는 경험을 반복했다.

**기능 1개는 괜찮다. 10개, 20개 쌓이면 무너진다.**

실제로 겪은 일들:
- 재사용 컴포넌트가 있는데 AI가 매번 새로 만든다
- 여러 모듈을 연계해서 써야 하는데, 컨텍스트 부족이나 오염으로 누락이 생긴다
- 기능을 추가할 때마다 기존 구조가 흔들리고, 어느 순간 전체를 다시 짜야 한다

왜 이런 일이 반복되는가? AI는 **"지금 이 기능"에만 집중**하기 때문이다. 전체 코드베이스를 한 번에 보지 못하고, 기존 구조를 탐색하는 것보다 새로 만드는 게 빠르면 새로 만든다. 프로젝트의 암묵적 규칙도 모른다. 그래서 기능이 쌓일수록 중복 코드가 늘고, 파일이 엉뚱한 곳에 생기고, 레이어 규칙이 하나씩 깨진다. 그리고 이게 20개, 30개 누적된 뒤에 발견하면? **AI가 만든 속도를 AI가 만든 기술 부채가 잡아먹는다.**

### 내가 시도한 해결: 도구 체인으로 아키텍처 지키기

**기존 코드가 곧 컨텍스트가 되어줄 것이고, 철저한 구조가 규칙이 되어줄 것이다** — 그래서 현업 프로젝트([Lamy-Work](https://github.com/LamySolution))에서 이런 도구들을 조합해서 아키텍처를 지켜왔다:

| 도구 | 역할 | 강제하는 것 |
|:---:|:---|:---|
| **ESLint + boundaries** | FSD 계층 강제 | 상위→하위 단방향 의존만 허용, 크로스 슬라이스 금지 |
| **dependency-cruiser** | 순환참조 방지 | 모듈 간 순환 의존 완전 금지, 계층 간 import 규칙 |
| **Semgrep** | 정적 분석 | Prisma는 infra에서만, IO 라이브러리는 infra에서만 |
| **커스텀 스크립트** | Public API 강제 | 모듈 간 import는 반드시 index.ts 통해서만 |

아키텍처는 지켜졌다. **하지만 이것도 문제가 있었다.**

- **느리다.** Husky pre-commit에 이 도구들을 전부 걸어놨는데, 커밋할 때마다 검사가 돌아가니 개발 속도에 확실한 마이너스였다.
- **도구 간 일관성이 떨어진다.** 같은 "infra에서만 Prisma를 쓴다"는 규칙을 ESLint boundaries로도 잡고, dependency-cruiser로도 잡고, Semgrep으로도 잡는다. 하나의 아키텍처 규칙을 여러 도구에 중복으로 표현해야 했다.
- **실시간이 아니다.** pre-commit 시점에 잡으면 이미 코드를 다 짠 뒤다. 코드를 작성하는 그 순간에 구조 위반을 알려줘야, AI든 사람이든 즉시 방향을 바꿀 수 있다.

docs 폴더에 엔지니어링 규칙 문서도 만들어봤다. 하지만 프로젝트를 개발하다 보면 실시간으로 변화가 생기는데, 문서가 그 속도를 따라가지 못했다. 오히려 이전 문서가 현재 코드와 맞지 않아서 개발에 방해가 되는 경우도 있었다.

프롬프트 하네스? 강제력 없다. CLAUDE.md? 참고 자료일 뿐이다. 규칙 문서? 코드가 바뀌면 금방 낡는다. 린터/훅? 느리고 사후 검증이다. **전부 "권고"이거나 "사후 처리"이거나 "금방 낡는 문서"다.**

아키텍처를 지키기 위해 만든 도구 체인이, 그 자체로 관리 비용이 되는 아이러니. 그래서 생각했다 — **이 모든 걸 프레임워크가 처음부터, 실시간으로 해주면 안 되나?** 프로젝트를 시작하는 순간부터 구조가 잡혀 있고, 별도 도구 조합 없이 하나의 Guard가 통합으로, 실시간으로 잡아주는 프레임워크. 그 고민의 결과물이 Mandu다.

### 왜 하필 웹 프레임워크인가

그렇다면 어디서 이 문제를 풀어야 하는가? 왜 프레임워크인가?

#### 이유 1: 웹이 AI 코딩의 최전선이다

프론트엔드 + 백엔드가 합쳐진 풀스택 웹 개발은 AI 코드 생성이 가장 활발한 영역이다. 동시에 레이어가 많고 파일이 많아서 아키텍처 붕괴가 가장 빈번하게 일어나는 곳이기도 하다. 문제가 가장 심한 곳에서 풀어야 한다.

#### 이유 2: 프레임워크는 강제력이 있다

프롬프트는 권고이고, 린터는 사후 검증이다. 하지만 프레임워크는 다르다. **구조를 벗어난 코드는 아예 동작하지 않는다.** Next.js에서 `app/` 바깥에 페이지를 만들면 라우팅이 안 되는 것처럼, 프레임워크의 규칙은 코드 레벨에서 강제된다. AI가 우회할 수 없다.

#### 이유 3: 피드백 루프가 즉각적이다

웹은 결과를 브라우저에서 바로 확인할 수 있다. Guard 위반 시 빌드가 실패하고, 그 이유가 터미널에 바로 뜬다. "만들고 → 검증하고 → 고치는" 루프가 자동으로 돌아갈 수 있는 환경이다.

**→ 프레임워크가 곧 하네스가 될 수 있다.**

### Mandu — "구조 자체가 API"인 Agent-Native 풀스택 프레임워크

> GitHub: [github.com/konamgil/mandu](https://github.com/konamgil/mandu)

이런 생각에서 출발한 것이 **Mandu**다.

기존 프레임워크(Next.js, Remix 등)가 **"사람이 작성하기 편한 것"**에 초점을 맞췄다면, Mandu는 **"사람이 설계(Spec)하고 AI가 구현(Filling)하되, 아키텍처가 무너지지 않도록(Guard) 하는 것"**에 집중한다.

한 줄로 요약하면:

> **"에이전트가 코딩해도 아키텍처가 무너지지 않는 개발 OS"**

#### 핵심 철학 — 5가지 불변 원칙

| 원칙 | 설명 |
|:---:|:---|
| **Filesystem-First** | `app/` 디렉토리가 SSOT. 파일 생성/삭제가 곧 라우트 정의이며, manifest는 파생 캐시일 뿐이다 |
| **Generated는 일회용** | `.mandu/generated/`는 언제든 재생성 가능. 꼬이면 "처음부터"가 아니라 "생성 다시" |
| **Slot에서만 작업** | AI는 `spec/slots/`, `spec/contracts/`에서만 비즈니스 로직 작성 |
| **Guard가 구조 보호** | Lint가 아니라 Guard가 아키텍처 위반을 실시간 차단 |
| **Self-Correction Loop** | 실패 시 Heal diff 제안 → 자동 재시도. 사람이 빠져도 루프가 돈다 |

#### 단방향 파이프라인

```
app/ (파일 생성 = 라우트 정의)
  ↓ FSScanner
routes.manifest.json (파생 캐시)
  ↓ Generator
.mandu/generated/ (일회용 코드)
  ↓ Guard (실시간 검증)
Dev Server / SSR (실행)
```

이 파이프라인에서 AI가 할 수 있는 일과 할 수 없는 일이 명확히 나뉜다:

| 영역 | AI가 할 수 있는 일 | AI가 할 수 없는 일 |
|:---:|:---|:---|
| **app/** | 페이지/API 라우트 파일 생성 | 프레임워크 규칙을 벗어난 구조 생성 |
| **spec/slots/** | 비즈니스 로직 자유롭게 작성 | Slot 바깥 코드 수정 |
| **Generated** | — | 생성된 코드 직접 수정 (재생성으로만 변경) |
| **Guard** | — | Guard 규칙 우회 |

#### MCP 서버 — AI와 프레임워크의 대화 채널

Mandu는 **15개 카테고리, 70개 이상의 MCP 도구**를 내장하여, AI 에이전트가 프레임워크와 직접 대화한다.

프롬프트로 "라우트 추가해줘"라고 하면 AI가 파일을 직접 만든다. 어디에, 어떤 형식으로 만들지는 AI 판단에 의존한다.

MCP 도구를 통하면? → 프레임워크가 정해진 위치에 정해진 형식으로 생성한다. **AI의 판단 여지를 줄이고, 정확성을 높인다.**

| 카테고리 | 주요 도구 | 역할 |
|:---:|:---|:---|
| Spec | `mandu_add_route`, `mandu_validate_manifest` | 라우트 추가/검증 |
| Guard | `mandu_guard_check`, `mandu_negotiate`, `mandu_guard_heal` | 구조 검증/협상/자동 수정 |
| Brain | `mandu_doctor`, `mandu_check_location`, `mandu_watch_start` | 아키텍처 감시/실시간 워치 |
| ATE | `mandu.ate.auto_pipeline`, `mandu.ate.heal` | 테스트 자동 생성/실패 복구 |
| Transaction | `mandu_begin`, `mandu_commit`, `mandu_rollback` | 변경사항 안전 관리 |
| Contract | `mandu_create_contract`, `mandu_generate_openapi` | API 스키마 정의/OpenAPI 생성 |
| Resource | `mandu.resource.create`, `mandu.resource.addField` | 데이터 모델 관리 |
| SEO | `mandu_preview_seo`, `mandu_seo_analyze` | SEO 메타데이터/분석 |

특히 **트랜잭션 API**가 핵심이다. AI가 여러 파일을 수정하다 중간에 실패하면? `rollback`으로 스냅샷 시점으로 완전 복원된다. 사람이 수습할 필요가 없다.

그리고 **Guard의 negotiate** 도구가 흥미롭다. AI가 "이 파일을 이 위치에 만들어도 되나?"를 Guard에게 먼저 물어보고, Guard가 허용/거부/대안을 제시한다. 코드를 짜기 전에 구조를 협의하는 것이다.
